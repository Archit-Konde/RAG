HTTP, the HyperText Transfer Protocol, is an application-layer protocol used to transfer data between clients and servers on the web. It defines how messages are formatted and transmitted, and how servers and browsers should respond to various commands. HTTP is the foundation of data communication on the World Wide Web and is used whenever a browser requests a web page, image, or other resource from a server.

HTTP follows a client-server model where a client, typically a web browser, sends a request to a server, which processes the request and returns a response. HTTP is a stateless protocol, meaning each request is completely independent and the server retains no memory of previous requests. This statelessness simplifies server design but requires mechanisms like cookies or sessions to maintain state across multiple interactions.

HTTP has evolved through several versions. HTTP/1.0 established the basic request-response cycle. HTTP/1.1 added persistent connections, chunked transfer encoding, and the required Host header. HTTP/2 introduced multiplexing, header compression, and server push, allowing multiple requests over a single connection. HTTP/3 replaced TCP with QUIC, a UDP-based protocol, reducing latency and improving performance on unreliable networks.

By default, HTTP communicates over port 80, while HTTPS, the secure version of HTTP that uses TLS encryption, communicates over port 443. These are well-known port numbers defined by IANA. Clients do not need to specify the port when connecting to a server on the default port. A custom port can be specified in the URL, for example http://example.com:8080/, and the server must be configured to listen on that port.

The GET method requests a resource from a server without modifying it. GET requests are safe, meaning they have no side effects on the server, and idempotent, meaning making the same request multiple times produces the same result. GET requests should not include a body. The response typically includes the requested resource in the body. Query parameters can be appended to the URL to filter or customize the response.

The POST method submits data to the server, typically to create a new resource or trigger a process. Unlike GET, POST requests include a body containing the data to be processed. POST is not idempotent: submitting the same POST request twice may create duplicate resources. Common uses include submitting HTML forms, uploading files, and sending JSON payloads to REST APIs. The server responds with a status code indicating success or failure.

The PUT method replaces an existing resource entirely with the data provided in the request body. If the resource does not exist, PUT may create it. PUT is idempotent: sending the same PUT request multiple times results in the same server state. Unlike POST, which typically appends or creates, PUT performs a full replacement. For partial updates, the PATCH method should be used instead of PUT.

The DELETE method requests that the server remove the specified resource. It is idempotent: deleting a resource that no longer exists returns a different status code, but the server state remains the same. A successful DELETE typically returns 200 OK with a response body, 204 No Content with no body, or 202 Accepted if the deletion is asynchronous. DELETE requests generally do not include a body.

The HEAD method is identical to GET but instructs the server to return only the response headers, not the response body. HEAD is used to check whether a resource exists, retrieve its metadata such as Content-Type and Content-Length, or verify that a resource has changed before downloading it. HEAD is safe and idempotent. It is useful for checking link validity and cache freshness without transferring the full resource.

The OPTIONS method requests information about the communication options available for a given resource or server. The server responds with an Allow header listing the HTTP methods it supports for that resource. OPTIONS is commonly used in CORS preflight requests, where the browser checks whether a cross-origin request is permitted before sending the actual request. OPTIONS is safe and idempotent and does not modify any server state.

The PATCH method applies a partial modification to an existing resource, unlike PUT which replaces the resource entirely. The request body contains a description of the changes to apply, such as a JSON Patch document or a partial JSON object. PATCH is not required to be idempotent, though implementations can be made idempotent. It is used when only certain fields need updating, reducing bandwidth compared to a full PUT replacement.

HTTP status codes in the 1xx range are informational responses indicating that the request was received and processing is continuing. The most common is 100 Continue, sent when the server acknowledges the request headers and the client should proceed to send the request body. 101 Switching Protocols indicates the server is switching to a different protocol as requested by the client, such as upgrading from HTTP to WebSocket.

HTTP 2xx status codes indicate that the request was received, understood, and accepted successfully. 200 OK is the standard response for a successful GET request. 201 Created is returned after a POST or PUT that creates a new resource. 204 No Content indicates success with no response body, often used for DELETE requests. 202 Accepted means the request was received but processing is not yet complete, used for asynchronous operations.

HTTP 3xx status codes indicate redirection, meaning the client must take additional action to complete the request. 301 Moved Permanently tells the client the resource has been permanently relocated to a new URL. 302 Found indicates a temporary redirect. 304 Not Modified tells the client the cached version is still valid and does not need to be re-downloaded. 307 Temporary Redirect preserves the HTTP method during redirection.

HTTP 4xx status codes indicate client errors, meaning the request was malformed or cannot be fulfilled. 400 Bad Request means the server cannot parse the request. 401 Unauthorized means authentication is required. 403 Forbidden means the server understood the request but refuses to authorize it. 404 Not Found indicates the resource does not exist. 405 Method Not Allowed means the HTTP method is not supported for that resource.

HTTP 5xx status codes indicate server-side errors. 500 Internal Server Error is the generic response when the server fails unexpectedly. 502 Bad Gateway occurs when the server receives an invalid response from an upstream service. 503 Service Unavailable indicates the server is temporarily unable to handle requests due to overload or maintenance. 504 Gateway Timeout means an upstream server failed to respond in time. Unlike 4xx errors, these indicate the fault lies with the server.

The Host header is the only header required in HTTP/1.1 requests. It specifies the domain name of the server and optionally the port number, allowing a single server to host multiple websites through virtual hosting. For example, Host: www.example.com tells the server which website the client is requesting. If the Host header is missing from an HTTP/1.1 request, the server must return a 400 Bad Request response.

The Content-Type header indicates the media type and encoding of the request or response body. Common values include text/html for HTML documents, application/json for JSON data, application/x-www-form-urlencoded for HTML form submissions, and multipart/form-data for file uploads. The header may also include a charset parameter, such as Content-Type: text/html; charset=UTF-8, specifying the character encoding of the body.

The Content-Length header specifies the exact number of bytes in the request or response body. The server uses this value to know when the body ends, especially over persistent connections where multiple requests share the same TCP connection. If Content-Length is absent and chunked transfer encoding is not used, the receiver cannot determine the body boundaries reliably. For GET and HEAD responses with no body, Content-Length is typically zero or omitted.

The Authorization header carries credentials for authenticating a client with a server. The most common scheme is Bearer token authentication, where a JWT or OAuth 2.0 access token is included, formatted as Authorization: Bearer <token>. Basic authentication encodes the username and password in base64, formatted as Authorization: Basic <encoded>. The server validates these credentials and returns 401 Unauthorized if they are missing or invalid.

The Cache-Control header specifies directives for caching behavior in both requests and responses. max-age=N instructs caches to store the response for N seconds. no-cache means the cache must revalidate with the server before using it. no-store means the response must not be stored in any cache. public allows shared caches to store the response, while private restricts caching to the browser only. must-revalidate requires stale responses to be revalidated before use.

The Accept header allows a client to specify which media types it is willing to receive, enabling content negotiation. For example, Accept: application/json tells the server the client prefers JSON. Multiple types can be listed with quality values, such as Accept: text/html,application/json;q=0.9, where q=0.9 indicates lower preference. The server selects the most suitable format and indicates it in the Content-Type response header.

The Set-Cookie header is sent by the server to instruct the client to store a cookie. It includes the cookie name and value, and optionally attributes like Expires, Max-Age, Domain, Path, Secure, and HttpOnly. The Secure flag ensures the cookie is only sent over HTTPS. The HttpOnly flag prevents JavaScript from accessing the cookie, reducing XSS risk. The client sends stored cookies in subsequent requests using the Cookie header.

Chunked transfer encoding allows the server to send a response body in pieces without knowing the total size in advance. Each chunk is prefixed with its size in hexadecimal followed by a carriage return and line feed. The transfer ends when a zero-length chunk is received. This is indicated by the Transfer-Encoding: chunked header in place of Content-Length. Chunked encoding is useful for streaming responses and dynamically generated content.

HTTP/1.1 introduced persistent connections, also called keep-alive, where a single TCP connection is reused for multiple HTTP requests and responses. This reduces the overhead of establishing a new connection for every request. By default in HTTP/1.1, connections are persistent unless Connection: close is explicitly sent. The Connection: keep-alive header can be used to confirm persistence. Persistent connections significantly improve performance for pages loading many resources.

Every HTTP request begins with a request line containing three components separated by spaces: the HTTP method, the request target, and the HTTP version. For example, GET /index.html HTTP/1.1 requests the index.html file using HTTP version 1.1. The request target is typically an absolute path and may include a query string. The request line is followed by headers, a blank line, and optionally a message body.

An HTTP response consists of a status line, headers, a blank line, and optionally a body. The status line includes the HTTP version, a three-digit numeric status code, and a human-readable reason phrase, for example HTTP/1.1 200 OK. Headers provide metadata about the response, such as Content-Type and Content-Length. The blank line separates the headers from the body, which contains the requested resource or an error message.

A URL, or Uniform Resource Locator, has several components. The scheme specifies the protocol, such as http or https. The host identifies the server domain name or IP address. The path specifies the resource location on the server. The query string, preceded by a question mark, contains key-value pairs for filtering or parameterizing the request. The fragment, preceded by a hash symbol, identifies a section within the document and is not sent to the server.

HTTP methods are classified by two properties: safety and idempotency. A safe method has no side effects on the server: GET, HEAD, and OPTIONS are safe. An idempotent method can be called multiple times with the same result: GET, HEAD, OPTIONS, PUT, and DELETE are idempotent. POST and PATCH are neither safe nor idempotent by default. These properties guide API design: idempotent methods are safer to retry on failures, and safe methods can be freely cached.

The ETag header provides a unique identifier, or entity tag, for a specific version of a resource. When a client caches a response containing an ETag, it can send the value back in an If-None-Match header on subsequent requests. If the resource has not changed, the server returns 304 Not Modified without a body, saving bandwidth. Similarly, the If-Modified-Since header enables conditional requests based on the last modification date of the resource.
